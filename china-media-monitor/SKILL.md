# China Media Monitor 技能

## 概述

本技能通过 `可用搜索工具` 逐一检索 5 大媒体矩阵（共 25 家中国权威媒体）的最新案例报道，
按用户指定的**题材范畴**筛选出**以真实案例为主**的报道（排除时评/理论/政策原文），
输出含原文链接的结构化每日简报。

---

### 场景 A：用户触发但未指定题材

→ 按 "0. 全覆盖" 处理。

## 执行流程（用户回复编号后）

### Step 1：解析用户选择
→ 解析编号列表（如 "1,4,6" → [T01, T04, T06]）
→ 若为 "0" → 全覆盖模式（每矩阵 5 篇尽量多题材均衡）
→ 若为关键词（如 "外来物种"）→ 按 `references/media-matrix.md` 末尾「题材自动归类」表匹配
→ 若附带矩阵/媒体限定 → 记录作为搜索范围

### Step 2：读取媒体矩阵配置
→ 读取 `references/media-matrix.md`，获取各矩阵媒体清单与优先级顺序
→ 读取主题词库，提取本次任务对应的题材关键词

### Step 3：按矩阵抓取候选文章（🌟 官网直取优先）

**抓取优先级（严格按此顺序，前一级成功才不执行下一级）**：

1. **官网要闻页直取**（首选）
   - 从 `references/media-matrix.md` 取每家媒体的"要闻入口"URL
   - 用 WebFetch / navigate 工具抓取该页面 HTML
   - 从页面中解析出今日/最近文章列表（标题、链接、日期）
   - 按题材关键词筛选命中的文章
   - 找够 5 篇即停，不再搜索

2. **官网首页兜底**（要闻页失败时）
   - 回退到该媒体根域名首页 `https://{domain}/`
   - 同样解析文章列表并筛选

3. **搜索引擎兜底**（前两级都失败时）
   - 用可用的搜索工具（可用搜索工具 / online-search / ProSearch 等，技能不绑定具体工具）
   - 读 `references/search-patterns.md` 按模板组合查询
   - 搜索词公式：`[媒体名] [题材核心词] [子题词] 案例 [年份]`

**硬性约束**：
- 每家媒体最多 **2 次** WebFetch + **2 次** 搜索查询，超过则放弃并降级到同矩阵下一家
- 每矩阵凑够 5 篇 OR 总尝试 15 次就停，不要死磕
- 同一篇文章不跨矩阵重复引用

### 🚨 媒体白名单（核心规则，违反立即作废该篇）

**只收录 `references/media-matrix.md` 明确列出的 25 家媒体的文章**。以下站点即使搜索返回、即使内容再优质，**一律丢弃不收录**：

❌ 封面新闻 (thecover.cn)、腾讯新闻 (news.qq.com / qq.com/rain)、今日头条、百家号 (baijiahao)、搜狐新闻、网易新闻、凤凰网、澎湃新闻（除非矩阵里明确列入）、新浪新闻、知乎、微博、微信公众号截图、地方小报自媒体、企业公众号

✅ 仅接受 25 家白名单媒体的官方域名，如 people.com.cn、news.cn、xinhuanet.com、cctv.com、farmer.com.cn、jfdaily.com 等（具体以 media-matrix.md 为准）

**识别方法**：
1. 拿到候选文章 URL 后，第一步查它的域名是否在 media-matrix.md 的 25 家列表里
2. 若正文标注"来源：人民日报"但 URL 来自聚合站 → 按人民日报收录（按来源判，不按 URL 判）
3. 若正文无来源标注且 URL 域名不在白名单 → **丢弃，不收录**

### 📊 媒体多样性配额（防止单一媒体刷屏）

- 单题材同一家媒体最多收录 **2 篇**，超过必须换家
- 每矩阵必须覆盖 **至少 3 家不同媒体**，凑不齐宁可该矩阵少于 5 篇，也不允许单家刷屏
- 写 md 前先看已落盘文章列表，统计各媒体已收数量，超标就跳过该篇去抓下一篇

### 🔍 搜索必须限定域名（禁止裸搜）

搜索兜底时，**查询词必须包含 `site:<该媒体官方域名>`**，防止搜索结果跑出白名单外的媒体。

示例：
- ✅ `site:people.com.cn 矿山修复 案例 2024`
- ✅ `site:news.cn 城市更新 老旧小区`
- ✅ `site:farmer.com.cn 高标准农田 田长制`
- ❌ `矿山修复 案例 2024`（裸搜会返回封面新闻、腾讯等非白名单媒体）

裸搜返回的结果一律视为无效，必须加 site: 重搜。

### 🗂 媒体/矩阵字段严格查表

输出 md 时的 `媒体` 和 `矩阵` 字段**禁止按文章内容推断**，必须从 media-matrix.md 查表：

- 媒体字段 = 正文"来源：XXX"标注的媒体名（优先）或白名单 URL 对应的媒体名
- 矩阵字段 = 该媒体在 media-matrix.md 中所属的矩阵编号（A/B/C/D/E 唯一确定）
- **禁止**同一家媒体在不同文章里被标成不同矩阵——这是 AI 瞎猜的典型信号

→ 保留：含 5W 要素（谁/何时/何地/做什么/结果）的真实案例报道

### 🚫 防卡死硬约束（必须遵守）

#### 🔒 方法锁定原则（最重要，违反必卡死）

- 第 1 个 md 用 Write 工具成功写入后，**后续所有 md 必须继续用 Write**，一篇一篇写
- 禁止在中途切换到"批量脚本"、"循环写入"、"write_file.py"、"先探测平台再批处理"等任何替代方案
- 即使要写 20 个文件，也是 **20 次 Write 调用**，不是 1 次批处理
- Write 调用可以在同一消息里并行发起多个（推荐），但禁止绕路到脚本
- 禁止说"需要用 XX 脚本写入"、"先探测平台"、"批量生成"这类论证性独白
- 禁止写完 md 后再 Read 回来"自我验证"——信任 Write 返回值

**原因**：AI 常在写完第 1 篇后产生"批量更高效"的念头，去找不存在的批处理工具，导致任务卡死。**重复的朴素动作 > 聪明的优化。**

---

落盘时**只允许用 Write 工具**写绝对路径 md 文件，例如：
```
Write(file_path="C:/Users/15403/Documents/每日新闻/T05_xxx.md", content="...")
```

**禁止行为**（违反会让任务卡死半天只出 1 个文件）：
- ❌ 不要用 PowerShell `Out-File` / `Set-Content` / `Add-Content`
- ❌ 不要用 `python3 -c "..."` 或外部脚本（write_file.py 等）写文件
- ❌ 不要先 `Test-Path` / `Get-ChildItem` / `ls` 探测目录是否存在——直接写
- ❌ 不要写临时文件到 `%TEMP%` 再移动
- ❌ **严禁"批量写入"思维**——不许先抓完所有媒体再统一写文件。这样会让单次输出内容爆量导致卡死

#### 🌀 严格串行模式（新增，防止上下文爆掉）

**每命中 1 篇文章 → 立即 Write 1 个 md 文件 → 播报进度 → 再去抓下一篇**

流程：
```
抓第 1 篇正文 → Write 第 1 个 md ✓ → 进度行 [1/N]
抓第 2 篇正文 → Write 第 2 个 md ✓ → 进度行 [2/N]
...
抓第 N 篇正文 → Write 第 N 个 md ✓ → 进度行 [N/N]
最后汇总阶段 → Write 1 个综述 md
```

**硬性要求**：
- 禁止在头脑里"攒"多篇文章正文后一次性写出——每篇正文拿到后 **下一个动作就是 Write**
- 综述文件放最后，且**不重复抄录各篇正文**，只放标题/链接/50字摘要/📄 文件路径
- 如某篇正文极长（>5000 字），分段写入时仍是 1 次 Write 搞定（用多段字符串拼接，不要拆多个文件）
- 每写完 1 个 md，聊天里输出一行 `✓ [i/N] 写入 xxx.md`，**让用户能看到进度推进**

**关键原因**：上一版"先抓完所有再批量写"会让 AI 把 20 篇正文都塞进工作记忆，再试图一次 Python 批处理，内容量超输出上限导致卡死。串行"抓一写一"把每次工作量压到最小。

**逐篇处理+进度播报流程**（推荐，便于用户看到卡在哪步）：

每处理一家媒体，在聊天中输出 4 行进度锚点（✓ 成功 / ✗ 失败 / ⏭ 降级）：

```
📋 {TXX} {题材名}｜共 5 家媒体待抓取

[1/5] 人民日报 → 访问要闻页... ✓ 找到 3 篇候选
      → 筛选命中 1 篇：《从一座矿山治理看...》
      → 抓取正文... ✓ (2834 字)
      → 写入 T05_生态_矿山修复_xxx(2025-03-06).md ✓

[2/5] 新华社 → 访问要闻页... ✓ 找到 5 篇候选
      → 筛选命中 0 篇（全时评/政策原文）
      → ⏭ 降级到搜索兜底... ✓ 找到 1 篇
      → 写入 ✓

[3/5] 农民日报 → 访问要闻页... ✗ 超时
      → ⏭ 降级到首页... ✓
      ...

[汇总] 5 家完成，落盘 4 篇 + 1 份综述 → C:/Users/15403/Documents/每日新闻/
```

**时间预算**（超限即停止当前媒体、降级到下一家）：
- 单家媒体：≤ 2 次 WebFetch + ≤ 2 次搜索 = 最多 4 次探测
- 单题材全流程：≤ 15 次抓取调用 + N 次 Write（N = 命中文章数）
- 总超时后：立即停止抓取，用已落盘的文章出短综述

### ⚠️ 媒体署名与日期识别（常见 Bug 修复）

**Bug 1：聚合站转载时署名搞错**
经常遇到 cfen.com.cn（中国改革报/市场报系）、news.cn、人民网地方频道等**聚合/转载站**。正文开头通常会注明"来源：XX"。

- ✅ **媒体字段填原始来源**（如正文写 "来源：人民日报"，则媒体 = 人民日报，不是 cfen.com.cn）
- ✅ 可在"矩阵"字段后加括号备注转载站，如 `矩阵：A（转载自 cfen.com.cn）`
- ❌ 不要把 URL 所在域名当成媒体名

识别规则：
1. 正文前 3 行扫描"来源：XXX"、"本报讯"、"（XX 日报记者 XXX）"这类署名
2. 若无来源标注，再回退到 URL 域名对应的媒体
3. 记者署名（作者：XXX）也要保留

**Bug 2：日期瞎编**
不要用"今天"或搜索元数据的日期。必须从 **URL 路径或正文顶部**提取真实发布日期。

- URL 常见格式：`.../202503/t20250306_xxx.html` → 2025-03-06
- 正文顶部格式：`2025-03-06 来源：人民日报`
- 页面元数据：`<meta name="publishdate" content="2025-03-06">`

**日期收录政策**（2026-04 更新）：
- ✅ **4 年内发布的案例都可收录**（当前日期往回推 4 年）
- ✅ 越新优先级越高：优先本周 > 本月 > 今年 > 近 2 年 > 近 4 年
- ❌ 不要强行造"今天日期"——宁可收录 2024 年的真实案例，也不编造 2026-04-18
- 文件名 `(YYYY-MM-DD)` 必须是**真实发布日期**

**必须生成两类 md 文件**：

**(1) 每篇文章一个独立 md 文件**
- 文件名格式：`{TXX}_{话题核心词}_{子题词}_{报道原标题}({YYYY-MM-DD}).md`
  - `核心词` / `子题词` 从 `references/media-matrix.md` 对应题材的词库中挑最贴合的
- 示例：`T02_民生_老旧小区改造_泸州枣林园小区改造后团年饭排起队(2026-04-03).md`
- 示例：`T06_城市更新_历史街区_上海800亿城市更新基金成立(2026-04-17).md`
- 内容：标题、题材标签、媒体名、发布日期、原文链接、50 字摘要、**完整原文正文**、案例类型副标签、关键词
- ⚠️ **必须抓取完整正文**：用 WebFetch 访问原文 URL 获取全文，不要写要点罗列或二次概括。若原文无法访问，明确标注"【正文未能获取】"
- 标题中的非法字符（`\ / : * ? " < > |`）替换为全角或下划线

**(2) 当日综述文件（每个题材一个）**
- 文件名格式：`综述_{YYYY-MM-DD}_{TXX}.md`
- 示例：`综述_2026-04-18_T02.md`
- 内容：抓取日期、题材、5 大矩阵简报条目（按模板）、跨矩阵 Top3、题材分布统计、所有落盘文章的相对路径索引

→ 读取 `assets/output-template.md`，套用具体字段模板
→ 文件写入后，在聊天中简要汇报："已写入 N 篇文章 + M 份综述到 C:/Users/15403/Documents/每日新闻/"，并列出文件清单
→ **不得**仅在聊天里输出条目后就结束——必须落盘

## 搜索执行规则

- 每次 可用搜索工具 查询字数控制在 **4~10 个字**
- 每个媒体矩阵至少执行 **3 次** 可用搜索工具（宽泛 → 精准 → 补充）
- 链接无法确认时，注明 `[链接待核实]` 并附搜索关键词
- 同一篇文章不跨矩阵重复引用
- 单题材：每矩阵 5 篇；多题材：按题材均匀分配；全覆盖：每矩阵 5 篇尽量多题材均衡
